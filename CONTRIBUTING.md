# Contributing to CacheMap

Thanks for your interest in improving CacheMap. This guide covers local setup,
the checks we run, and how to add a new analysis rule.

## Setup

```bash
git clone https://github.com/kingsleychenlab/cachemap
cd cachemap
npm install
```

Run the CLI in development without building:

```bash
npm run dev -- analyze --offline .github/workflows/ci.yml
```

## Checks

Every change must pass all four, cross-platform (Linux, macOS, Windows) on
Node 20 and 22:

```bash
npm run lint        # ESLint, zero warnings
npm run typecheck   # strict tsc, no emit
npm test            # vitest
npm run build       # tsc -> dist/ and esbuild -> action/index.js
npm run format:check
```

`npm run lint:fix` and `npm run format` apply automatic fixes.

## Adding an analysis rule

1. Create `src/analysis/<rule>.ts` exporting a function
   `(input: AnalysisInput) => Finding[]`. Use `makeFinding` for deterministic
   ids and one of `inferredSavings` / `measuredSavings` / `unknownSavings` for
   any time claim — **never** invent an exact number or a percentage confidence.
2. Register it in `src/analysis/runner.ts`.
3. Add fixtures under `test/fixtures/workflows/` and tests under `test/`.
4. Document it in `docs/RULES.md` using the rule id as the heading (the SARIF
   `helpUri` links there).

### Rule guidelines

- Prefer deterministic, evidence-backed conclusions. When a value is dynamic or
  cannot be resolved statically, preserve the expression and avoid unsupported
  conclusions.
- Security issues are `kind: 'security'` and never carry savings.
- Do not recommend removing something that has a real dependency (e.g. a
  `needs:` justified by an artifact download).

## The bundled Action

`action/index.js` is generated by `npm run build`. If you change action or
library code, **rebuild and commit** the bundle — CI verifies it is up to date.

## Commit & PR

- Keep changes focused; add tests for new behaviour.
- Update `CHANGELOG.md` under `Unreleased`.
- CacheMap analyzes its own workflows in CI; keep them clean.

## Releasing (maintainers)

1. Update `version` in `package.json` and `src/version.ts`, and move the
   `CHANGELOG.md` `Unreleased` section under the new version.
2. `npm run build && npm test`.
3. Tag `vX.Y.Z` and push; the release workflow publishes to npm and updates the
   major-version tag used by the Action.
